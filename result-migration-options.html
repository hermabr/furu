1. The problem, concretely

You add a knob to a spec and a corresponding field to its result:

@dataclass(frozen=True)
class TrainResult:
    accuracy: float
    final_model: Path | None      # NEW result field

class TrainRun(Spec[TrainResult]):
    dataset: str
    save_final_model: bool = True  # NEW spec field — True for new runs

    def create(self) -> TrainResult: ...

Three requirements, one of which furu cannot express today:

  1. Old runs should map to save_final_model=False — that is what they behaviorally were. Already solved: Added("save_final_model", default=False) pins the historical value independently of the field's dataclass default (the existing pin semantics).
  2. New runs default to True — the dataclass default, unrelated to the migration pin. Already works.
  3. Old stored results must decode as the new TrainResult, i.e. gain final_model=None. Not expressible today. This document is about requirement 3.
Scoping assumption (per current furu semantics — no result hashing or code tracing yet): the same schema + fields always produce the same result. Therefore a result only ever needs migrating when the spec schema changes, i.e. result changes always ride along a spec migration chain. A "result-only" migration has no trigger point — a direct manifest hit bypasses migration resolution entirely.

2. What happens today (and where it breaks)

Migration never rewrites result bytes. When an old stored spec, replayed forward through the migrations chain, equals the current spec, furu writes a small result-link.json pointing at the old base dir (src/furu/migration/links.py:137 — result_dir_for_loading) and loads the old result/manifest.json verbatim.

The old manifest for the example above looks like:

{
  "$furu": {
    "|kind": "dataclass",
    "|class": "myproj.train.TrainResult",
    "|fields": { "accuracy": 0.93 }
  }
}

Decoding is strict: _load_validated_fields (src/furu/result/bundle.py:482) raises on missing or extra fields against the current TrainResult. So the spec-side migration succeeds, the link is written — and then the load blows up because final_model is absent. The gap is precisely a hook that transforms the manifest JSON between "link followed" and "value decoded".

3. Three independent design axes

The options below mostly differ along three orthogonal choices. Worth separating them, because you can mix and match:

AxisChoices
Where declared attached to the spec step that caused it (A, B) · a standalone step in the chain (C) · on the result type itself (D)
What the transform sees raw manifest JSON with $furu wrappers · a "logical fields" view with wrappers hidden · decoded Python objects (infeasible — the old result shape no longer has a type to decode into)
When applied lazily, in memory, every time a link is followed · eagerly materialized once into the new base dir (§8)

4. Option A — per-step transform function your idea #1

Every step type grows an optional result= callable that receives the stored result manifest and returns the migrated one.

def _add_final_model(manifest: JsonValue) -> JsonValue:
    # caller must know the $furu wrapper encoding
    inner = dict(manifest["$furu"])
    inner["|fields"] = {**inner["|fields"], "final_model": None}
    return {"$furu": inner}

class TrainRun(Spec[TrainResult]):
    dataset: str
    save_final_model: bool = True

    migrations = (
        Added("save_final_model", default=False, result=_add_final_model),
    )

Answering "all fields or only Added/Retyped?": there is no reason to restrict — the kwarg is orthogonal to which spec change occurred (a Renamed or Rewrite can equally coincide with a result-shape change), so put it on every step type uniformly. Only breaking=True steps should reject it: results before a breaking cut are superseded, so a transform there is dead code → declaration-time TypeError, consistent with the existing Added(breaking=True, default=…) contradiction check.

Pros

  • One mechanism covers everything, including value-dependent rewrites (e.g. computing the new field from old result values).
  • Smallest API surface: one kwarg on the existing step dataclasses.
  • Change is declared exactly where its cause is — the step reads as "this spec change implies this result change".

Cons

  • User handles the raw $furu/|kind/|fields wrapper encoding — currently an internal format with no public contract. Nested dataclasses, artifacts, tuples/sets all have non-obvious encodings.
  • Nothing validatable at class creation: a typo'd result field name only explodes at load time, far from the declaration (spec steps get declaration-time TypeErrors today; this would be a regression in feel).
  • Opaque to migration_path — the link audit trail can only record "result=<function _add_final_model>".
  • Not hashable/serializable — a liability once you add result hashing or provenance for migrations later.
  • Common case (add one constant field) costs ~6 lines of wrapper surgery.

5. Option B — declarative result_changes= your idea #2 recommended

Mirror the spec-step vocabulary with a small set of frozen result-step dataclasses, attached to the spec step that caused them:

class TrainRun(Spec[TrainResult]):
    dataset: str
    save_final_model: bool = True

    migrations = (
        Added(
            "save_final_model",
            default=False,
            result_changes=(ResultAdded("final_model", value=None),),
        ),
    )

The vocabulary (all @dataclass(frozen=True, slots=True), exported next to the spec steps):

ResultAdded(field, *, value)      # insert; `value` is any spec-serializable value,
                                  # encoded with the existing to_json — user never
                                  # sees the $furu wrapper format
ResultRenamed(field, *, to)       # move a key
ResultRemoved(field)              # drop a key
ResultRewrite(fn)                 # escape hatch, see below

The escape hatch operates on a logical fields view — the top-level result fields with wrappers already unwrapped one level (same trick as the existing spec-side Rewrite's _SourceFields wall, which rejects unknown keys) — and receives the already-migrated spec fields for value-dependent cases:

def _fill(fields: Mapping[str, JsonValue], *, spec: Mapping[str, JsonValue]) -> Mapping[str, JsonValue]:
    # e.g. derive the new field from old result/spec values
    return {**fields, "final_model": None}

migrations = (
    Added("save_final_model", default=False,
          result_changes=(ResultRewrite(_fill),)),
)

What this buys, mechanically:

  • Declaration-time validation. Spec[T] already names the result type. When T is a dataclass or pydantic model, validate_migration_declaration can walk result_changes in reverse exactly like it walks spec steps today — checking that ResultAdded("final_model", …) targets a real field of the new T, renames chain correctly, etc. (Skipped when T is dict/opaque.)
  • Audit trail. migration_path in result-link.json gets honest entries: "ResultAdded('final_model', value=None)".
  • Future-proof for hashing. Declarative steps are canonically serializable; when result hashing / code tracing arrives, these can participate in identity. Functions can't.
  • The 90% case is one line, and the value round-trips through to_json, so ResultAdded("checkpoint", value=Path("…")) or a nested dataclass value just works.

Pros

  • Users never touch the internal manifest encoding.
  • Validated at class creation, matching the existing DX of spec steps.
  • Introspectable, describable, hashable.
  • Escape hatch (ResultRewrite) keeps full power available without making it the default path.
  • Same "attached to the causing step" locality as Option A.

Cons

  • More API surface: 4 new exported classes + a kwarg on 5 step types.
  • Two-level vocabulary (spec steps containing result steps) is a little more to learn than one function.
  • v1 scope question: dotted paths for nested result fields ("trainer.final_model") — probably defer (§11).

6. Option C — standalone result steps in the chain

Result steps live as siblings of spec steps in the same migrations tuple, associated by position:

migrations = (
    Added("save_final_model", default=False),
    ResultAdded("final_model", value=None),   # applies to the same boundary
)

Pros

  • Flat, uniform chain — no nesting; reads chronologically.
  • No kwarg added to existing step types.

Cons

  • Generation ambiguity. _build_chain reconstructs historical schemas by un-applying steps in reverse; a step with no spec effect creates two adjacent generations with identical spec schema. That's the same class of problem as the "dead Retyped" that furu already rejects as a declaration error. Fixable by special-casing result-only steps as boundary annotations rather than generation creators — but at that point it is Option B with a different surface syntax and a weaker (positional) association.
  • Footgun. A chain containing only result steps never fires (direct manifest hit skips resolution — see the scoping assumption). Must be a declaration error, which is awkward to explain.
  • Cause and effect are linked only by adjacency; a reordered tuple silently changes meaning.

7. Option D — lenient decode via result-type defaults

No migration API at all: make the strict result loader fill dataclass defaults for keys missing from the stored manifest.

@dataclass(frozen=True)
class TrainResult:
    accuracy: float
    final_model: Path | None = None   # loader fills this when absent in old bundles

class TrainRun(Spec[TrainResult]):
    dataset: str
    save_final_model: bool = True
    migrations = (Added("save_final_model", default=False),)   # spec side only

Pros

  • Zero new migration concepts; the additive case (the most common one) is free.
  • No changes to links or resolution.

Cons

  • Not generation-scoped. The default fills any missing key, forever, in any bundle — including ones missing the field due to corruption or a buggy codec. It converts the strict loader's best property (loud failure on shape drift) into silent repair.
  • Conflates two meanings of a default: "value for old runs" vs. "value for the result type." They coincide in this example but not in general (a rename or computed backfill has no default-shaped answer).
  • Can't express renames, removals, or value-dependent rewrites.
  • No audit trail — migration_path never learns the result was reshaped.
  • Forces the result type to carry a default the domain may not want (a fresh run should arguably be required to set final_model explicitly).

Could still be worth having later as an explicit opt-in (result_decode = "fill-defaults") for quick-and-dirty projects, but it is not a substitute for a real mechanism.

8. Execution axis: lazy transform vs. materialization orthogonal to A–D

Lazy (recommended)

When result_dir_for_loading resolves via a link, compose the result_changes of every step from the link's start generation onward and apply them to the manifest JSON in memory, before decode. Old bytes stay canonical and untouched — preserving furu's "superseded data is never deleted or rewritten" invariant. The transforms must stay in code for as long as links exist, but that is already true of the spec steps themselves.

Eager materialization

On first link-follow, write the transformed manifest into the new base dir so subsequent loads are direct hits, then drop the steps from code eventually.

Materialization is tempting but expensive in invariants: the bundle's artifacts/ payloads are referenced by paths relative to the bundle dir, so you'd need to copy or hardlink potentially-huge artifacts (or teach the manifest cross-dir references); concurrent writers need the same atomic-rename dance as _store_result; and a buggy transform leaves persistently wrong materialized bytes rather than a fixable in-memory view. Defer until there's a demonstrated load-time cost.

10. Recommendation

Option B, applied lazily, with ResultRewrite as the escape hatch inside the same tuple. It matches furu's existing design DNA — declarative frozen steps, declaration-time validation, human-readable migration_path, escape hatch as a named step (Rewrite already sets this precedent on the spec side) — and it is the only option that stays hashable for the day result identity becomes content-addressed.

The full worked example under the recommendation:

@dataclass(frozen=True)
class TrainResult:
    accuracy: float
    final_model: Path | None

class TrainRun(Spec[TrainResult]):
    dataset: str
    save_final_model: bool = True          # new runs: True

    migrations = (
        Added(
            "save_final_model",
            default=False,                 # old runs were behaviorally False
            result_changes=(ResultAdded("final_model", value=None),),
        ),
    )

# old stored run:  TrainRun(dataset="c10")                          [pre-migration]
# ------------------------------------------------------------------
# TrainRun(dataset="c10", save_final_model=False)
#   → links to the old result dir; manifest gains final_model=None
#     in memory at load; decodes as the new TrainResult.
# TrainRun(dataset="c10", save_final_model=True)
#   → no source matches → computes fresh; create() writes final_model
#     itself. The result change never applies to fresh runs.

Design decisions folded in:

  • result_changes is accepted on all five step types (it's orthogonal to the kind of spec change), rejected only on breaking=True steps (dead code by definition — declaration error).
  • value= is encoded with the existing to_json, so any spec-serializable value works and the wrapper format stays private.
  • ResultRewrite(fn) sees logical top-level result fields behind the same "wall" object the spec-side Rewrite uses (unknown-key access raises, key-set changes beyond declared adds/removes raise), plus the migrated spec fields as a keyword for value-dependent backfills.
  • Changes compose along the chain in order, one sub-tuple per boundary, exactly like spec steps.

11. Implementation touchpoints

  1. src/furu/migration/steps.py — add ResultAdded / ResultRenamed / ResultRemoved / ResultRewrite dataclasses and a ResultChange alias; add result_changes: tuple[ResultChange, ...] = () to the five step types; extend validate_migration_declaration to (a) reject result_changes with breaking=True, (b) when T is introspectable, replay result changes in reverse against T's field set the same way spec fields are replayed today.
  2. src/furu/migration/resolution.py — _build_chain records per-boundary encoded ResultAdded values (mirror of added_defaults); _describe_step includes result changes in the step description.
  3. src/furu/migration/links.py — _ResultLink gains the source generation's start index (so the loader knows which boundaries' result changes to compose without re-running resolution). Old link files lacking the field: fall back to re-resolving once and rewriting the link.
  4. src/furu/result/bundle.py — load_result_bundle accepts an optional manifest transform applied between json.loads and _load_value; a small helper maps logical field ops onto the top-level wrapper (dataclass/pydantic |fields, or a plain dict manifest).
  5. src/furu/execution/load_or_create.py — the load path threads the composed transform through when (and only when) the dir came from a link.
  6. Cache note: _RESOLUTION_CACHE keys on (type, storage) — result changes live on the class, so no invalidation change needed, but tests should pin that editing result_changes alone still produces a new chain description.

12. Open questions

  • Nested result fields. Should ResultAdded("trainer.final_model", …) support dotted paths into result-embedded dataclasses? Defer; ResultRewrite covers it meanwhile.
  • Result-embedded dataclass evolution. If a dataclass embedded inside results (with a |class marker) gains a field, every parent result containing it fails to decode. The spec side solves the analogous problem with cascading child chains; the same machinery could eventually cascade result changes. Out of scope here, but the declarative vocabulary is the representation that would make it possible.
  • Result-type identity. Today the schema hash covers spec fields; if T itself ever enters the hash, a result-shape change would also move the base dir and Option C's "no trigger" footgun disappears. Worth deciding before result hashing lands.
  • Materialization later? If link-follow transform cost ever matters (huge manifests, hot loops), add opt-in materialization behind the same declarative steps — the API doesn't change.
